# Архитектура системы промптов

## Почему именно так

Эта система — ответ на пять болей которые я нашёл у себя (и ты наверное тоже):

### Боль 1: "GEMINI.md превратился в монстра"
**Симптом:** GEMINI.md = 500+ строк. Агент помнит первые 50 и последние 30, середину игнорирует. Когда добавляешь новое правило — оно теряется в шуме.

**Решение:** **L1 жёстко ограничен по объёму.** Каждый файл L1 = 30-80 строк. Если не влезает — значит это не глобальное правило, а проектное. Иди в L2.

### Боль 2: "Правила размазаны по 10 файлам"
**Симптом:** Правило про тесты — в GEMINI.md. Правило про commit messages — в CONTRIBUTING.md. Правило про error handling — где-то в docs/. Не помнишь где что.

**Решение:** **Доменная структура L1.** 4 файла, каждый про свою тему. Хочешь найти правило про стек — знаешь что это STACK.md. Про тон — STYLE.md. Всегда знаешь куда положить новое правило.

### Боль 3: "Каждый раз формулирую задачу по-разному"
**Симптом:** "Сделай фичу X" — сегодня одно, завтра другое, послезавтра третье. Агент каждый раз работает чуть иначе. Качество плавает.

**Решение:** **Skills = шаблоны процедур.** Один раз написал skill `feature.md` — теперь при любом "сделай X" агент вызывает один и тот же skill и работает одинаково. Качество стабильное.

### Боль 4: "Перешёл с Cursor — потерял всё"
**Симптом:** `.cursorrules` = 9KB специфики для Cursor. Перешёл на Antigravity — начинай с нуля.

**Решение:** **Vendor-neutral содержимое.** Текст правил — одинаковый. Конвенция папки — разная (`.agent/` vs `CLAUDE.md` vs `.cursorrules`). Переезд = копирование + переименование папки.

### Боль 5: "Агент игнорирует правила"
**Симптом:** "Пожалуйста, пиши тесты" — агент забывает. "Не забудь про edge cases" — забывает. Правила soft, легко игнорируются.

**Решение:** **Skills — это enforced процедуры.** Skill `feature.md` говорит: "шаг 4: написать unit-тесты. Шаг 5: написать integration тесты. Шаг 6: прогнать test suite." Не "пожалуйста напиши тесты" — а "обязательный шаг, иначе skill не complete".

## Три уровня — подробнее

### L1: Глобальный (загружается ВСЕГДА)

**Где:** `.agent/*.md` (Antigravity) / `CLAUDE.md` (Claude Code) / `.cursorrules` (Cursor)
**Когда:** IDE читает автоматически при старте сессии
**Объём:** ~5KB суммарно, кратко
**Содержимое:** то что верно для ВСЕХ твоих проектов навсегда

| Файл | Что внутри | Когда редактировать |
|---|---|---|
| `IDENTITY.md` | Кто ты, твой уровень, твои ограничения | Когда меняется твоя роль |
| `PRINCIPLES.md` | Неизменные правила: что делать, что нет | Раз в год, очень редко |
| `STACK.md` | Какие технологии используешь по умолчанию | Когда меняешь стек |
| `STYLE.md` | Как общаешься с агентом: тон, язык, формат | Когда надоедает старый тон |

### L2: Проектный (для конкретного проекта)

**Где:** `project/*.md` в репо проекта
**Когда:** Читается автоматически если лежит в `project/`, или ты руками говоришь "прочитай project/PROJECT.md"
**Объём:** ~10-30KB суммарно
**Содержимое:** то что верно для ЭТОГО проекта, но не для всех

| Файл | Что внутри |
|---|---|
| `PROJECT.md` | Что за проект, зачем, для кого, метрики успеха |
| `ARCHITECTURE.md` | Архитектура этого проекта: компоненты, потоки данных, границы |
| `CONVENTIONS.md` | Code style, naming conventions, запрещённые паттерны |

### L3: Сессионный (на одну сессию)

**Где:** `session/YYYY-MM-DD-краткое-имя.md`
**Когда:** Ты создаёшь утром через `./scripts/new-session.sh "Цель"`
**Объём:** ~1-3KB
**Содержимое:** что делаем конкретно СЕГОДНЯ

```markdown
# Сессия: 2026-06-30 — OAuth Google

## Цель
Добавить вход через Google в AI OFM dashboard. Юзеры жалуются на пароли.

## Задачи
- [ ] Создать bead bd-014 "OAuth Google"
- [ ] Использовать skill `feature` для имплементации
- [ ] Тесты обязательны (skill `feature` шаг 4-5)
- [ ] Деплой в dev (skill `ship`)

## Ограничения
- НЕ ломать существующий credentials login
- НЕ менять NextAuth core, только добавить провайдер
- Все env vars — в Secret Manager, не .env
```

## Skills — каталог переиспользуемых процедур

Каждый skill = файл `.agent/skills/<name>.md` с фиксированной структурой:

```markdown
# Skill: <название>

## Когда вызывать
Триггеры: "сделай", "почини", etc.

## Что делает
Краткое описание.

## Процедура (шаги)
1. Шаг 1
2. Шаг 2
3. ...

## Выход
Что skill должен произвести.
```

**6 базовых skills покрывают 90% работы:**

| Skill | Триггер | Что делает |
|---|---|---|
| `brainstorm` | "давай сделаем X" / "хочу X" | Socratic-вопросы → spec → понятно что строить |
| `feature` | "сделай фичу" / "implement" | spec → plan → bead → TDD → review → ship |
| `debug` | "не работает" / "баг" / "X падает" | reproduce → isolate → fix → test → regression check |
| `review` | "проверь код" / "review this" | code review по чеклисту |
| `refactor` | "почисти" / "рефакторь" / "extract" | план рефакторинга → маленькие шаги → тесты остаются зелёными |
| `ship` | "задеплой" / "release" | smoke tests → deploy → verify → announce |

## Vendor-neutral — почему это работает

Содержимое файлов L1/L2/Skills — **plain markdown**, не специфика IDE. Разница только в КОНВЕНЦИИ ПАПКИ:

| IDE | L1 | L2 | Skills |
|---|---|---|---|
| **Antigravity** | `.agent/*.md` | `project/*.md` | `.agent/skills/*.md` |
| **Claude Code** | `CLAUDE.md` или `.claude/*.md` | `docs/project/*.md` | `.claude/skills/*.md` |
| **Cursor** | `.cursorrules` | `docs/project/*.md` | `.cursor/rules/*.mdc` |

Текст внутри — копируешь один-в-один. Меняется только путь.

Когда переезжаешь с Antigravity на Claude Code — `./scripts/migrate-to-claude.sh` копирует файлы в новые пути. 30 секунд.

## Анти-паттерны (как делать НЕ надо)

### ❌ Один файл на 500 строк
```
GEMINI.md:
# Project X
- rules
- more rules
- style
- conventions
- tools
- ...
```
→ Забудь про половину. Не ищи нужное.

### ✅ Структурированные маленькие файлы
```
.agent/
├── IDENTITY.md      (50 строк)
├── PRINCIPLES.md    (80 строк)
├── STACK.md         (40 строк)
└── STYLE.md         (30 строк)
```
→ Нашёл сразу. Добавил новое правило — знаешь куда.

### ❌ "Пожалуйста, пиши тесты"
```
GEMINI.md: "Не забывай писать тесты"
```
→ Агент забывает.

### ✅ Skill с обязательными шагами
```markdown
## Skill: feature

### Процедура
1. ...
2. ...
3. ...
4. **ОБЯЗАТЕЛЬНО**: написать unit-тесты
5. **ОБЯЗАТЕЛЬНО**: написать integration тесты
6. **ОБЯЗАТЕЛЬНО**: прогнать `pytest` / `vitest`, все зелёные
7. ...
```
→ Skill не complete пока все ОБЯЗАТЕЛЬНЫЕ шаги не сделаны.

### ❌ Промпт "сделай как в прошлый раз"
→ У агента нет памяти между сессиями.

### ✅ SESSION_BRIEF с конкретными задачами
```markdown
## Задачи
- [ ] OAuth Google (skill `feature`)
- [ ] Smoke tests (skill `ship`)
```
→ Агент читает, делает.

### ❌ Vendor-locked правила
```
.cursorrules: "Use Cmd+K for..."
```
→ Перешёл на Antigravity — потерял.

### ✅ Vendor-neutral правила
```
.agent/STYLE.md: "Используй keyboard shortcut для..."
```
→ Работает везде.

## Как мигрировать существующий эволюционный GEMINI.md

1. Создать `.agent/` папку
2. Разбить GEMINI.md на 4 файла:
   - Что про меня → `IDENTITY.md`
   - Что про правила → `PRINCIPLES.md`
   - Что про технологии → `STACK.md`
   - Что про тон/формат → `STYLE.md`
3. То что про конкретный проект → `project/PROJECT.md` и др.
4. Skills, если есть повторяющиеся задачи — вынести в `.agent/skills/`
5. Старый GEMINI.md — удалить (или оставить как legacy если страшно)

30-60 минут работы. После — больше НЕ возвращаешься к 500-строчному монстру.
