# Skill: refactor

## Когда вызывать
- "почисти код"
- "рефакторь X"
- "extract Y в отдельный модуль"
- "убери дублирование"
- (перед большим PR — превентивно)

## Цель
Улучшить структуру кода **без изменения поведения**. Тесты должны остаться зелёными до и после.

## Процедура

### 1. **ОБЯЗАТЕЛЬНО**: baseline тесты
Прежде чем трогать код — запусти полный test suite.
```bash
npm run test  # запомни: все зелёные
```

Если что-то красное — **не рефакторь**. Сначала почини. Иначе не поймёшь что ты сломал.

### 2. Определить scope рефакторинга
Не "почисти всё". Конкретно:
- "Вынести X в отдельный модуль"
- "Убрать дублирование между Y и Z"
- "Переименовать A → B потому что путает"
- "Заменить callback-based X на async/await"

**ЗАПРЕЩЕНО**: менять поведение под видом рефакторинга. Это отдельный таск.

### 3. План маленьких шагов
Разбей на 2-5 минимальных коммитов. Каждый коммит:
- Меняет одну вещь
- Не ломает тесты
- Легко откатить если что

Пример плана:
```
1. Extract helper function (коммит 1)
2. Move helper to lib/ (коммит 2)
3. Update imports across project (коммит 3)
4. Rename for clarity (коммит 4)
```

### 4. Делай по одному шагу
После каждого:
```bash
npm run test  # всё ещё зелёные?
npm run lint
npm run typecheck
```

Если красное — откати (`git checkout .`) и переосмысли.

### 5. **ОБЯЗАТЕЛЬНО**: поведение НЕ изменилось
- Если API изменился → это breaking change, нужна ADR + major version bump. Не делай под видом refactor.
- Если функция начала возвращать другое → это bug или feature, не refactor.
- Если добавил новую логику → это feature.

### 6. Проверить coverage не упал
```bash
npm run test:coverage  # должен быть ≥ того что был до
```

Если упал — где-то не покрыл новый код тестом.

### 7. PR с `refactor:` типом
```bash
git checkout -b refactor/<short-name>
# ... мелкие коммиты ...
git log --oneline  # проверь что коммиты осмысленные

git push -u origin refactor/<short-name>
gh pr create --title "refactor(<scope>): <description>"
```

В PR description ОБЯЗАТЕЛЬНО:
- Что именно рефакторится
- Почему (какая боль)
- Что НЕ изменилось (поведение)
- Как проверено что поведение не сломалось (tests)

## Выход
- Поведение идентично до и после
- Тесты зелёные, coverage ≥ baseline
- 2-5 осмысленных коммитов
- PR с `refactor:` типом и явным "what didn't change"
- Мне: что улучшилось и почему безопасно

## Когда прерваться
- Если нужно менять > 100 файлов → слишком большой рефакторинг, разбить на несколько PR
- Если в процессе нашёл баг → создай bead, не чини в refactor PR
- Если рефакторинг требует менять API → стоп, нужен brainstorm + ADR
- Если 2 итерации тесты красные → откати всё, переосмысли план

## ЗАПРЕЩЕНО
- "Заодно поправил баг" в refactor PR — отдельный PR
- Менять naming везде сразу — делай rename как отдельный коммит
- Touch файлы "потому что тут тоже некрасиво" — вне scope
- Большие коммиты на 500+ строк — разбивай
- "Потом разберусь" — сейчас или bead
