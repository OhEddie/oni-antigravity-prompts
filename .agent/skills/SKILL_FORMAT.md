# Формат Skill

Каждый skill в `.agent/skills/*.md` имеет одинаковую структуру. Это позволяет агенту **одинаково** работать с любым skill.

## Структура

```markdown
# Skill: <name>

## Когда вызывать
Триггеры в моих промптах. Например:
- "сделай X" / "implement X" / "build X"
- "X не работает" / "X падает" / "debug X"

## Цель
Что skill должен произвести. 1-2 строки.

## Процедура (шаги)
1. Шаг 1 (что делать)
2. Шаг 2
3. **ОБЯЗАТЕЛЬНО**: что-то критичное (без этого skill не complete)
4. Шаг 4
5. **ЗАПРЕЩЕНО**: что нельзя делать
6. Шаг 6

## Выход
- Что skill создаёт/изменяет в репо
- Что должен сказать мне в итоге

## Когда прерваться и спросить
- Условия когда не делать самому, а прийти ко мне.
```

## Принципы написания

### 1. Шаги конкретные, не абстрактные

❌ "Сделай хорошо"
✅ "Написать unit-тесты покрывающие happy path + 2 edge case"

### 2. ОБЯЗАТЕЛЬНЫЕ шаги — явно помечены

❌ "Не забудь про тесты"
✅ "**ОБЯЗАТЕЛЬНО**: `npm run test` должен быть зелёный"

### 3. ЗАПРЕЩЕНО — тоже явно

❌ "Не делай плохих вещей"
✅ "**ЗАПРЕЩЕНО**: коммитить секреты. **ЗАПРЕЩЕНО**: force push в main."

### 4. "Когда прерваться" — это страховка

Не все решения может принять агент. Если встретил — пусть спросит, а не угадывает.

❌ Везде "если сомневаешься — спроси"
✅ Конкретный список: "если меняешь публичный API → спроси. Если добавляешь новую зависимость → спроси. Если трогаешь миграции → спроси."

### 5. Выход — что должно быть на диске

Не абстрактно "сделано", а конкретно:
- "Создан файл X"
- "Изменён файл Y (строки Z-W)"
- "Bead bd-NNN обновлён → status done"
- "PR создан → URL"
