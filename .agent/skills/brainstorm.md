# Skill: brainstorm

## Когда вызывать
- "давай сделаем X"
- "хочу X"
- "я думаю про X"
- "нужно X, но непонятно как"
- "обсудим X"

## Цель
Превратить расплывчатую идею в **чёткую спецификацию**, которую можно начать делать. Не писать код. Не делать. Понять.

## Процедура

### 1. Понять контекст
- Прочитай `.agent/IDENTITY.md` (кто я)
- Прочитай `project/PROJECT.md` (что за проект, зачем)
- Если есть `session/SESSION_BRIEF.md` — прочитай
- Если есть Beads — `bd ready` (может, мы это уже начали)

### 2. Socratic-вопросы (5-7 штук)
Задай по одному за раз, дождись ответа. Не пачкой.

**Области вопросов:**
1. **Проблема:** что болит? у кого? как сейчас решается?
2. **Границы:** что ВНУТРИ scope, что СНАРУЖИ?
3. **Пользователи:** кто основной? какие edge-кейсы?
4. **Данные:** что храним? откуда берём? кто меняет?
5. **Метрики успеха:** как поймём что сделано хорошо?
6. **Риски:** что может пойти не так? что будет если проект умрёт через год?
7. **Альтернативы:** "почему не сделать Y?" — это часто даёт лучшие ответы чем "как сделать X"

### 3. Сформулировать spec (3-5 блоков)
Когда вопросов хватает, покажи spec чанками по 100-200 слов:

```markdown
## Spec: <feature name>

### Проблема
[1 абзац]

### Что строим
[1 абзац — что появится в системе]

### Что НЕ строим
[явные out-of-scope]

### Acceptance criteria
- [ ] ...
- [ ] ...

### Открытые вопросы
- [если есть]
```

### 4. Получить sign-off
Спроси: "Spec ок? Можно делать?"

Если есть замечания — обнови. Если новые вопросы — вернись к шагу 2.

### 5. Создать bead (если medium+large)
```bash
bd create "<feature name>" -p P1 -d "<full description>"
bd update <id> --status open
```

**ЗАПРЕЩЕНО на этом этапе**: писать код, менять файлы, делать коммиты. Только spec.

## Выход
- Spec показан и согласован
- Bead создан (если medium+large)
- Готов перейти к skill `feature`

## Когда прерваться и спросить
- Если 2 итерации вопросов не сходятся — спроси: "может, я неправильно понял задачу?"
- Если spec противоречит существующему коду — обсуди со мной, не правь молча
