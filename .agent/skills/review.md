# Skill: review

## Когда вызывать
- "проверь код"
- "review this"
- "посмотри PR"
- "что думаешь про X?"
- (автоматически — перед каждым PR через skill `feature` step 6)

## Цель
Найти проблемы **до** merge. 2-stage: сначала соответствие спеке, потом качество кода.

## Процедура

### Stage 1: Spec compliance

Проверь **соответствие спеке / acceptance criteria**:
- [ ] Все требования из spec выполнены?
- [ ] Ничего лишнего из out-of-scope?
- [ ] Edge cases покрыты (не только happy path)?
- [ ] Acceptance criteria выполнены (бизнес-логика, не технические детали)?

**Если spec нет** → скажи мне: "Нужен spec. Сначала `brainstorm`?"

### Stage 2: Quality review

**Структура:**
- [ ] Функции ≤ 50 строк, файлы ≤ 300 строк
- [ ] Нет deep nesting (> 3 уровней)
- [ ] Нет copy-paste (DRY)
- [ ] Naming понятный (не `data1`, `temp`, `foo`)
- [ ] Imports сортированы

**Логика:**
- [ ] Error handling не swallow exceptions
- [ ] Нет `as any` / `# type: ignore` без комментария почему
- [ ] Async правильное (нет fire-and-forget где нельзя)
- [ ] Нет race conditions
- [ ] Нет memory leaks (subscriptions не закрыты)

**Безопасность:**
- [ ] Нет SQL injection (все queries parameterized)
- [ ] Нет user input в shell (`shell=True` без экранирования)
- [ ] Нет секретов в коде / комментариях / логах
- [ ] Auth check на каждом protected endpoint
- [ ] Input validation (zod/pydantic) на каждом user-facing endpoint

**Тесты:**
- [ ] Тесты есть
- [ ] Coverage ≥ 70% для нового кода
- [ ] Happy path + edge cases + error case
- [ ] Тесты независимы (можно запускать в любом порядке)
- [ ] Тесты быстрые (< 5 сек unit tests)

**Производительность:**
- [ ] Нет N+1 query
- [ ] Нет блокирующих операций в async путях
- [ ] Нет утечек в циклах

**Документация:**
- [ ] README обновлён если API изменился
- [ ] OpenAPI / schema обновлена
- [ ] Комментарии объясняют ЗАЧЕМ (не ЧТО)
- [ ] Если архитектурное решение — есть ADR

**Если найдено нарушение PRINCIPLES.md** → блокируй merge, скажи мне.

## Как выдавать вердикт

Формат:
```markdown
## Review

### Verdict
✅ Approve | ⚠️ Approve with comments | ❌ Request changes

### Stage 1 — Spec compliance
✅ Все критерии выполнены
- [если что-то не выполнено — перечисли]

### Stage 2 — Quality
**Структура:** ✅ / ⚠️ / ❌ + детали
**Логика:** ✅ / ⚠️ / ❌ + детали
**Безопасность:** ✅ / ⚠️ / ❌ + детали
**Тесты:** ✅ / ⚠️ / ❌ + детали
**Производительность:** ✅ / ⚠️ / ❌ + детали

### Конкретные замечания
1. `lib/auth.ts:42` — race condition между verify и invalidate. Нужно сериализовать.
2. `__tests__/avatar.test.ts:15` — нет edge case для null. Добавить.
3. `app/api/v1/users/route.ts:8` — N+1 query, сделать JOIN.

### Хорошо
- Понятный naming
- Хорошее покрытие тестами
- ...
```

## Выход
- Verdict: ✅ / ⚠️ / ❌
- Конкретные замечания с file:line
- Мне в чат: "Review готов. Verdict: ❌. 3 замечания. [ссылка на full review]"

## Когда прерваться
- Если PR > 1000 строк → скажи "слишком большой, разбей на несколько"
- Если вижу что меняется публичный API без ADR → блокируй
- Если вижу что-то подозрительное в безопасности → блокируй немедленно, обсуди

**ОБЯЗАТЕЛЬНО**: каждое замечание должно иметь file:line. Без конкретики — это не review, а шум.

## ЗАПРЕЩЕНО в review
- "Looks good!" без реальной проверки
- Комментировать стиль ("тут можно переписать красивее") — это не bug, noise
- Делать fix самому — только репортить. Пусть автор (агент) правит.
