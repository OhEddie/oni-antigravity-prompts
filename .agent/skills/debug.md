# Skill: debug

## Когда вызывать
- "X не работает"
- "X падает"
- "получаю ошибку Y"
- "X работает локально но не в проде"
- "тесты красные"

## Цель
Найти root cause и починить. Не угадывать. Не патчить симптом.

## Процедура

### 1. **ОБЯЗАТЕЛЬНО**: воспроизвести
Прежде чем чинить — **получи стабильное воспроизведение**.
- Локально: запусти, получи ошибку
- В проде: открой Sentry, найди stack trace, получи id юзера + timestamp
- Если не воспроизводится → спроси меня: "когда последний раз работало? что изменилось?"

### 2. Собрать факты (НЕ теории)
- Stack trace (полный, не первый фрейм)
- Что в логах? (stdout, Sentry, browser console)
- Какой последний коммит когда работало?
- Какие env vars / конфиг отличаются от рабочего состояния?

### 3. Сформулировать гипотезы (2-3 штуки)
Не "что-то сломалось". А конкретно:
- "Гипотеза 1: N+1 query на /api/v1/users потому что нет include"
- "Гипотеза 2: race condition в WebSocket reconnect"
- "Гипотеза 3: env var PRODUCTION_API_KEY не проброшен в Cloud Run"

### 4. Проверить гипотезы (от дешёвой к дорогой)
1. **Read code** — посмотри как написано
2. **Add logging** — добавь `logger.debug` если мало инфы
3. **Run in isolation** — минимальный repro без всего остального
4. **Inspect state** — открой БД, проверь данные
5. **Inspect env** — `env | grep X` / `printenv`

**ЗАПРЕЩЕНО**: угадывать и патчить наугад. Если 2 гипотезы обе возможны — спроси.

### 5. Найти root cause
Когда нашёл — **объясни в одну строку**: "Баг: в `lib/auth.ts:42` race condition между `verify()` и `invalidate()`, второй запрос использует устаревший token."

### 6. Написать failing test **ДО** фикса
```typescript
test('should reject expired token in concurrent request', async () => {
  // setup: два запроса с одним token, второй когда первый уже истёк
  // expect: второй получает 401
});
```

Убедись что тест падает именно на баге.

### 7. Минимальный фикс
**ЗАПРЕЩЕНО**: рефакторить всё подряд пока там. Только то что чинит баг.

### 8. Убедиться что тест проходит + остальные не сломались
```bash
npm run test  # все зелёные
npm run lint
npm run typecheck
```

### 9. Проверить regression
- Другие места где этот код используется — все ещё работают?
- Edge cases — все ещё корректные?
- Если фикс меняет поведение → обнови тесты + spec

### 10. PR с `fix:` типом
```bash
git checkout -b fix/<short-name>
# ... фикс ...
git commit -m "fix(<scope>): <description>

Root cause: <one liner>
Closes: #<issue> or bd-NNN

Refs: bd-NNN (regression if found)"
gh pr create --title "fix(<scope>): <description>"
```

## Выход
- Root cause найден и объяснён
- Failing test → passing test (TDD)
- Минимальный фикс
- Regression проверен
- PR создан с `fix:` типом
- Bead (если был) → done

## Когда прерваться и спросить
- Не могу воспроизвести за 3 попытки → стоп, спроси
- Root cause требует менять архитектуру → стоп, нужен brainstorm + возможно ADR
- Баг в чужой библиотеке → обсуди: фиксить у себя / форкнуть / ждать апдейт
- Множественные баги одновременно → разбить на несколько beads + PRs

## ЗАПРЕЩЕНО
- "Починил, наверное" без проверки
- Fix-ception (фикс поверх фикса без понимания почему)
- `try { ... } catch { }` без логирования (swallow exception — скрываешь баг)
- `as any` / `# type: ignore` чтобы заткнуть type checker (маскируешь реальную проблему)
- "TODO: проверить позже" — создай bead или сделай сейчас
